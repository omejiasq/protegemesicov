1. Resumen Ejecutivo

PROTEGEMESICOV es una plataforma SaaS multi-tenant desarrollada para empresas de transporte vigiladas por la Superintendencia de Transporte de Colombia. Permite la gestión integral de flotas de vehículos: alistamientos diarios, mantenimientos preventivos y correctivos, control de conductores y proveedores, y reporte automático de datos al sistema SICOV (Sistema de Información de Calidad del Operador de Transporte) mediante la API oficial de Supertransporte.

La plataforma está construida sobre una arquitectura de microservicios en contenedores Docker, con 7 APIs NestJS independientes, un frontend web en Vue 3, una aplicación móvil en Flutter, almacenamiento de objetos con MinIO, caché con Redis y base de datos MongoDB Atlas Enterprise con replicación y respaldos automáticos.

El sistema se encuentra actualmente en producción en el dominio https://sicov.protegeme.com con certificado SSL, alojado en un servidor VPS de SSDNodes. La hoja de ruta contempla la migración a Oracle Cloud Infrastructure (OCI) en una segunda fase, incorporando firewall perimetral, separación de cargas de trabajo y mejoras de seguridad significativas.

Aplicación móvil iOS/Android que consume directamente las APIs REST de api-maintenance y api-auth para registrar alistamientos y mantenimientos en campo.

5. Flujos de Integración y Carga de Datos

5.1 Sincronización Automática con SICOV (Supertransporte)

El módulo SicovSyncModule dentro de api-maintenance ejecuta un proceso Cron cada hora que procesa una cola de operaciones pendientes (sicov_sync_queue) y las envía a la API oficial de Supertransporte usando las credenciales (vigiladoId + vigiladoToken) de cada empresa.

5.2 Ingesta Externa — API para Terceros

El módulo ExternalIngestionModule permite que empresas con sistemas propios (ERP, software de taller) envíen datos a la plataforma mediante una API Key por empresa sin necesidad de autenticación JWT. Soporta dos modos:

  • Tiempo real: un registro por petición (POST /external/enlistment, POST /external/preventive, POST /external/corrective).
  • Batch: array de registros en una sola petición (POST /external/batch/enlistments, etc.). El sistema retorna el resultado de cada ítem individualmente.

5.3 Carga desde App Móvil (Flutter)

La app móvil consume directamente las APIs REST con JWT. El flujo típico de un conductor en ruta es:

  1. Login → obtiene JWT con enterprise_id embebido.
  2. Selecciona vehículo asignado → consulta catálogo de ítems de alistamiento (habilitados para su empresa).
  3. Completa el formulario de alistamiento (checklist de 11 códigos SICOV) → POST /enlistments.
  4. La plataforma encola automáticamente el reporte a Supertransporte.
  5. Opcionalmente registra mantenimiento preventivo o correctivo desde el taller.

6. Base de Datos — MongoDB Atlas Enterprise

¿Por qué MongoDB Atlas Enterprise y no Community?

La plataforma utiliza MongoDB Atlas Enterprise (tier M10 o superior), lo que aporta garantías críticas para una plataforma regulada como SICOV:

Característica Atlas EnterpriseValor para PROTEGEMESICOV
Replica Set de 3 nodos (primario + 2 secundarios) Alta disponibilidad: si cae el nodo primario, la elección automática garantiza continuidad en <30 segundos. Sin pérdida de datos.
Backups automáticos cada hora (Continuous Cloud Backup) Punto de recuperación (RPO) de máximo 1 hora. Frente a un borrado accidental de alistamientos o mantenimientos, se puede restaurar al estado exacto anterior al incidente.
Cifrado en reposo (Encryption at Rest) Los datos sensibles de conductores, licencias y operaciones cumplen los requisitos de protección de datos exigidos por la normativa colombiana (Ley 1581 Habeas Data).
Cifrado en tránsito (TLS obligatorio) Toda conexión entre los microservicios y MongoDB viaja cifrada, incluso dentro de la misma región de la nube.
Network Peering / IP Allowlist Solo las IPs del servidor de producción tienen acceso al cluster. Mitiga ataques de fuerza bruta a la base de datos.
Atlas Search (Lucene integrado) Disponible para búsquedas avanzadas en historial de mantenimientos y conductores sin necesidad de un motor de búsqueda externo.
Performance Advisor y Auto-Indexing Sugiere índices automáticamente basado en los patrones de consulta reales, optimizando los filtros más usados (por enterprise_id + active, placa, etc.).
Auditoría de base de datos Log completo de operaciones de lectura/escritura. Requerido para demostrar cumplimiento ante Supertransporte.

7. Gestión de Archivos — MinIO

MinIO es un servidor de almacenamiento de objetos compatible con la API de Amazon S3, desplegado como contenedor Docker en el mismo host. Gestiona todos los archivos binarios de la plataforma:

  • Logos de empresas
  • Imágenes de vehículos
  • PDFs generados de alistamientos (formato configurable por empresa)
  • PDFs de mantenimientos preventivos y correctivos
  • Documentos adjuntos (SOAT, RTM, licencias)

El frontend accede a los archivos a través de un endpoint proxy (/files/proxy?url=...) que agrega el token de autorización, evitando exponer directamente las credenciales de MinIO al navegador.

Riesgo actual: MinIO almacena sus datos en un volumen Docker montado en el disco local del VPS (./data/minio). Si el servidor falla sin respaldo, los archivos se perderían. Ver recomendación en Sección 11 (Fase 2 OCI).

8. Caché — Redis 7

Redis se despliega como contenedor Docker (imagen redis:7-alpine) dentro de la misma red interna. Se utiliza para reducir la latencia en consultas frecuentes y repetitivas:

  • Catálogos de alistamiento y mantenimiento: los ítems configurados por empresa (raramente cambian) se cachean para evitar roundtrips a MongoDB en cada petición de la app móvil.
  • Listados de vehículos activos por empresa: útiles en los formularios de selección del operador.
  • Resultados de dashboard: métricas agregadas con TTL de minutos para no recalcular en cada refresh.
  • Rate limiting: contadores de peticiones por API Key de terceros para prevenir abuso.
Estado actual: Redis no tiene configuración de persistencia (appendonly) ni contraseña (requirepass) en el entorno de desarrollo. En producción debe activarse autenticación y, en la Fase 2, migrarse a Oracle Cache con OCI o Redis gestionado.

10. Seguridad — Estado Actual y Riesgos

Medidas de seguridad implementadas:
  • Comunicación HTTPS con SSL/TLS activo en el dominio
  • Autenticación JWT con expiración configurable
  • Contraseñas hasheadas con bcrypt (salt 10)
  • Aislamiento de datos por enterprise_id en todos los endpoints
  • API Keys con hash para sistemas de terceros
  • MongoDB Atlas con IP Allowlist y TLS obligatorio
  • Validación de correo único por usuario para recuperación de contraseña segura
  • Índices sparse-unique en campos sensibles (usuario, correo)
  • Forzado de cambio de contraseña en primer login (must_change_password)
  • Control de empresas activas/inactivas con bloqueo de login
Riesgos identificados en la infraestructura actual:
  • Sin firewall de red (WAF/NGFW): los puertos de los microservicios podrían estar expuestos directamente a Internet sin filtrado. Un escaneo con Nmap revelaría todos los servicios.
  • Sin IDS/IPS: no hay detección de intrusiones ni bloqueo de ataques de fuerza bruta a nivel de red.
  • Redis sin contraseña: si el puerto 6379 es accesible externamente, Redis puede ser comprometido.
  • MinIO en disco local: sin respaldo externo independiente del VPS. Una falla de disco implica pérdida total de archivos.
  • Sin backup del sistema de archivos: solo MongoDB tiene respaldo automático (Atlas). Los archivos en MinIO y los volúmenes Docker no tienen política de respaldo.
  • Certificado Let's Encrypt: gratuito pero con vencimiento cada 90 días. Requiere monitoreo de la renovación automática.

11.1 Componentes Recomendados en OCI

🛡️ 1. Firewall y Protección Perimetral

OCI WAF (Web Application Firewall) frente al Load Balancer. Protege contra OWASP Top 10: inyección SQL, XSS, CSRF, bots, DDoS en capa 7. Reglas predefinidas para APIs REST. Complementar con Security Lists y Network Security Groups para bloquear todo tráfico que no venga del Load Balancer hacia la subred privada.

☸️ 2. OCI Kubernetes Engine (OKE) — Separación de Cargas

En lugar de un único VPS, se recomienda migrar los microservicios a OKE (Kubernetes gestionado por Oracle). Beneficios:

  • Escalado horizontal automático (HorizontalPodAutoscaler): si el tráfico de alistamientos aumenta en hora pico, api-maintenance escala automáticamente.
  • Rolling deployments sin downtime.
  • Aislamiento de servicios: cada microservicio en su propio namespace con Network Policies.
  • Secrets cifrados con OCI Vault (no más archivos .env en disco).

Alternativa más económica (si el presupuesto es limitado): dos VMs OCI Compute: una para el backend (todos los microservicios en Docker Compose) y otra para el almacenamiento (MinIO → Object Storage), con un Load Balancer delante y Security Groups configurados correctamente. Este enfoque ya resuelve el problema del firewall y la separación de datos.

🗄️ 3. OCI Object Storage en lugar de MinIO local

Se recomienda reemplazar MinIO por OCI Object Storage, que es:

  • Compatible con la API S3: el código no necesita cambios, solo cambiar las credenciales y el endpoint.
  • Durabilidad 99.999999999% (11 nueves): Oracle garantiza redundancia geográfica automática.
  • Sin gestión de disco: no hay volúmenes que mantener ni riesgo de llenado de disco.
  • Pre-Authenticated Requests: generación de URLs firmadas para acceso temporal a archivos (evita el proxy actual).
  • Lifecycle Policies: archivado automático de PDFs viejos a OCI Archive Storage (costo ~10x menor).
  • Integrado con OCI WAF: las descargas de archivos también pasan por el firewall.

⚡ 4. Redis Gestionado (OCI Cache)

OCI Cache es Redis Enterprise gestionado por Oracle. Elimina la necesidad de administrar el contenedor, activa persistencia (AOF), replicación activo-pasivo y failover automático. También permite configurar contraseña y TLS desde la consola OCI sin tocar configuración de Docker.

🔑 5. OCI Bastion — Acceso Administrativo Seguro

En lugar de exponer SSH en la IP pública del servidor, OCI Bastion provee un túnel SSH efímero auditado. El acceso al servidor requiere autenticación MFA en la consola OCI antes de abrir el túnel. Todos los accesos quedan registrados con usuario, fecha y hora.

🔒 6. OCI Vault — Gestión de Secretos

Los secretos actuales están en archivos .env en el servidor. En OCI Fase 2, todas las cadenas de conexión (MongoDB URI, JWT Secret, MinIO keys, SICOV tokens) se almacenan en OCI Vault con HSM (Hardware Security Module). Los microservicios los recuperan en tiempo de arranque via API, sin que existan en disco ni en el repositorio de código.

📊 7. Observabilidad — APM + Logging

OCI Application Performance Monitoring + OCI Logging Analytics permiten:

  • Trazas distribuidas entre microservicios (detección de cuellos de botella).
  • Centralización de logs con búsqueda en tiempo real.
  • Alertas proactivas: si api-maintenance supera 2 segundos de latencia o la cola SICOV acumula más de 100 pendientes, se envía notificación por email/SMS.
  • Dashboard de salud del sistema en tiempo real.

12. Recomendaciones para Arquitectura IoT de Escala

PROTEGEMESICOV está posicionada como una plataforma de gestión de flota. La siguiente evolución natural es incorporar telemetría en tiempo real desde los vehículos (IoT): localización GPS, acelerómetros, sensores de motor (OBD-II), consumo de combustible. Estas son las recomendaciones desde la perspectiva de arquitectura IoT:

12.1 Protocolo de Comunicación para Dispositivos IoT

  • MQTT sobre TLS (puerto 8883): protocolo estándar industrial para IoT. Consumo de batería mínimo. El dispositivo GPS/OBD del vehículo publica en un topic como fleet/{enterprise_id}/{vehicle_id}/telemetry. Un broker MQTT (e.g., EMQX o HiveMQ) gestiona miles de conexiones concurrentes.
  • No usar REST para telemetría continua: una petición HTTP por cada dato de GPS (cada 5 segundos × 100 vehículos = 1.200 req/min) satura la API. MQTT mantiene una conexión persistente por dispositivo.

12.3 Recomendaciones Adicionales de Arquitectura para Escala IoT

API Gateway

Con múltiples microservicios expuestos, se recomienda introducir un API Gateway (Kong, AWS API Gateway o el Load Balancer de OCI con routing por path) para: centralizar autenticación, rate limiting por empresa, versionado de API, y reducir la superficie de ataque (solo el gateway tiene IP pública).

Event-Driven Architecture para Alistamientos Masivos

Cuando múltiples conductores cargan alistamientos simultáneamente (hora pico antes de salida de rutas), el uso de una cola de mensajes (OCI Queue / RabbitMQ) desacopla la recepción del procesamiento y la sincronización SICOV, evitando timeouts y mejorando la experiencia del conductor. El patrón SicovSyncQueue ya implementado en la plataforma es el primer paso en esta dirección.

Caché Multi-Nivel

Además de Redis para datos calientes, implementar caché a nivel del API Gateway (TTL corto para endpoints de catálogos) y caché en el cliente móvil (Flutter Hive) para funcionamiento offline en zonas sin cobertura.

Base de Datos de Series Temporales

Los datos de GPS, velocidad y métricas de motor crecen exponencialmente con el tiempo. MongoDB no está optimizado para este tipo de datos. Se recomienda InfluxDB o TimescaleDB (extensión de PostgreSQL) para telemetría, manteniendo MongoDB para los documentos de negocio (alistamientos, mantenimientos, empresas).

Autenticación de Dispositivos IoT

Cada dispositivo GPS debe autenticarse con un certificado X.509 único (no usuario/contraseña). OCI IoT Hub o EMQX soportan autenticación mutual TLS (mTLS) con PKI, garantizando que solo dispositivos autorizados publiquen datos de telemetría.

Cumplimiento Normativo y Datos de Conducción

Los datos de telemetría de conductores (velocidad, frenadas, rutas) pueden estar sujetos a la Ley 1581 de Habeas Data en Colombia. Se recomienda implementar políticas de retención (e.g., purga automática de telemetría de GPS después de 12 meses), cifrado de identificadores de conductor en la capa de telemetría y consentimiento explícito del conductor al inicio del turno.
