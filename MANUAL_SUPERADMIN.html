1. El Rol SuperAdmin

1.1 ¿Qué puede hacer el SuperAdmin?

El SuperAdmin es el nivel más alto de acceso en ProtegeMeSICOV. A diferencia del administrador de empresa (Admin), el SuperAdmin tiene visibilidad y control sobre todas las empresas registradas en el sistema.

Capacidades exclusivas del SuperAdmin:
  • Crear, editar, activar y desactivar empresas vigiladas.
  • Configurar las credenciales SICOV (VigiladoId y Token) de cada empresa.
  • Asignar el tipo de paquete a cada empresa.
  • Crear el primer usuario administrador de una empresa nueva.
  • Acceder a todas las funciones de un administrador de empresa.
  • Usar herramientas avanzadas de diagnóstico y reparación SICOV.
  • Gestionar API Keys para integraciones externas.

1.2 Acceso al Sistema

1
Acceda a la URL del sistema.
2
Ingrese las credenciales del usuario SuperAdmin (usuario y contraseña de nivel global).
3
Será redirigido al Dashboard. El menú mostrará una sección adicional: ADMINISTRACIÓN.
🔒 Seguridad crítica: Las credenciales SuperAdmin deben ser custodiadas con el mayor nivel de seguridad. No las comparta ni las use en dispositivos sin control. Habilite siempre el cierre de sesión al terminar.

1.3 Menú del SuperAdmin

El SuperAdmin ve todas las secciones del menú, incluyendo la sección exclusiva:

2. Gestión de Empresas

Acceda desde: ADMINISTRACIÓN → Empresas.
Es el módulo central del SuperAdmin. Permite gestionar el ciclo de vida completo de cada empresa vigilada registrada en la plataforma.

2.1 Listar Empresas

Al ingresar verá una tabla con todas las empresas. Las columnas son:

ColumnaDescripción
NombreRazón social de la empresa vigilada.
NITNúmero de Identificación Tributaria.
PaqueteTipo de plan asignado (enterprise u otros).
VigiladoIdIdentificador de la empresa en el sistema SICOV de Supertransporte.
Token VigiladoToken de autenticación para la API de Supertransporte.
EstadoActiva o Inactiva.
AccionesEditar, Activar/Desactivar.

2.2 Crear una Empresa

1
Haga clic en el botón Nueva empresa.
2
Complete los datos básicos:
  • Nombre: Razón social de la empresa.
  • NIT: Sin dígito de verificación, sin puntos ni guiones.
  • Tipo de paquete: Seleccione el plan contratado.
3
Configure las credenciales SICOV (ver sección 2.5).
4
Haga clic en Guardar. La empresa quedará activa inmediatamente.
5
Cree el primer usuario administrador para esa empresa (ver sección 10 del Manual de Empresa).

2.3 Editar una Empresa

Haga clic en el ícono de edición (lápiz) de la empresa que desea modificar. Se abrirá un formulario con todos los campos editables. Actualice los datos necesarios y haga clic en Guardar.

⚠️ Atención: Cambiar el VigiladoId o el Token Vigilado de una empresa afecta directamente su capacidad de sincronizar con SICOV. Solo realice estos cambios si recibió nuevas credenciales de Supertransporte.

2.4 Activar / Desactivar una Empresa

En la lista de empresas, haga clic en el ícono de toggle (activar/desactivar). El sistema pedirá confirmación antes de proceder.

  • Desactivar: Los usuarios de esa empresa no podrán iniciar sesión. Los datos se conservan.
  • Activar: Restaura el acceso completo de la empresa al sistema.
⚠️ Impacto operativo: Al desactivar una empresa, todos los usuarios de esa empresa perderán el acceso inmediatamente. Use esta función solo en casos justificados (fin de contrato, suspensión por incumplimiento, etc.).

2.5 Configuración de Credenciales SICOV

Cada empresa vigilada necesita dos credenciales para comunicarse con la API de Supertransporte:

CampoDescripciónEjemplo
VigiladoId NIT de la empresa tal como está registrado en SICOV Supertransporte. Es el identificador principal de la empresa vigilada ante el ente regulador. 891100299
Token Vigilado Token de autenticación proporcionado por Supertransporte para el acceso a su API. Se incluye en cada llamada como cabecera de autorización. eyJhbGciOiJ...
ℹ️ Origen de las credenciales: Estas credenciales son suministradas directamente por la Superintendencia de Transporte a cada empresa vigilada. El SuperAdmin las ingresa al registrar o actualizar la empresa en el sistema.

2.6 Tipo de Paquete

El campo Tipo de paquete permite clasificar las empresas según el plan contratado:

PaqueteDescripción
enterprisePlan empresarial completo con todas las funciones habilitadas.
basicPlan básico (funciones limitadas según configuración).

4. Auditoría SICOV

Acceda desde: Reportes → Auditoría SICOV.
El SuperAdmin puede auditar el historial completo de comunicaciones con la API de Supertransporte para cualquier empresa activa en la sesión actual.

4.1 Consultar el Reporte de Auditoría

Use los filtros disponibles:

  • Desde / Hasta: Rango de fechas de las operaciones.
  • Operación: Tipo de llamada (guardarAlistamiento, guardarMantenimiento, login, etc.).
  • Estado: Todos / Exitosos / Con error.

Haga clic en Buscar para aplicar los filtros.

4.2 Interpretar Resultados

ColumnaDescripción
FechaFecha y hora exacta de la operación.
OperaciónNombre de la función llamada (ej: guardarAlistamiento).
HTTPCódigo de respuesta HTTP. 201/200 = éxito en la capa de red. Ojo: SICOV puede retornar 201 con error embebido.
EstadoOK o ERROR. El sistema detecta errores embebidos en respuestas 2xx.
TiempoMilisegundos que tardó la operación.
Mensaje SupertransporteTexto literal de la respuesta de SICOV. En errores indica la causa exacta.
ℹ️ Sobre errores embebidos: La API de SICOV a veces devuelve un código HTTP 200 o 201 pero con un campo interno status: 404 o status: 500 en el cuerpo de la respuesta. El sistema los detecta automáticamente y los registra como ERROR en la auditoría.
ℹ️ Seguridad: Los campos sensibles como tokens y contraseñas se enmascaran automáticamente antes de guardarse en la auditoría.

4.3 Corregir Registros Históricos

Si registros antiguos no aparecen al filtrar por empresa, use el botón "Corregir registros históricos". Este proceso asigna el enterpriseId a registros que fueron creados antes de que se implementara ese campo.

El sistema mostrará cuántos registros fueron actualizados.

5. Sincronización SICOV — Herramientas Avanzadas

Esta sección describe las herramientas disponibles a nivel técnico (API) para gestionar, diagnosticar y reparar la sincronización con SICOV. Estas operaciones se realizan directamente sobre el endpoint /sicov-sync del microservicio api-maintenance.

5.1 Estados de Sincronización

EstadoValor en BDSignificado
SINCRONIZADOsyncedEnviado y aceptado por SICOV. Tiene fechaSyncSicov asignada.
PENDIENTEpendingEn cola de reintentos. SICOV no estaba disponible al momento del registro.
FALLIDOfailedExcedió el máximo de reintentos. Requiere intervención manual.
DEMOdemoRegistro de prueba creado con API Key en modo demo. No se envía a SICOV.

5.2 Cron de Reintentos Automáticos

El sistema ejecuta automáticamente un proceso de reintentos cada hora mediante un cron job integrado en el microservicio api-maintenance (usando @nestjs/schedule).

  • Busca todos los registros con sicov_sync_status: 'pending'.
  • Para cada registro, verifica primero si ya fue enviado a SICOV (para evitar duplicados).
  • Si no existe en SICOV, lo envía. Si ya existe, lo marca como synced.
  • Si el envío falla de nuevo, incrementa el contador de reintentos.

Para forzar la ejecución inmediata del cron (útil en urgencias):

POST /sicov-sync/trigger-batch

5.3 Diagnóstico de Placa en SICOV

Permite consultar directamente en SICOV maestras el historial de mantenimientos registrados para una placa específica. Útil para verificar si los envíos llegaron correctamente.

GET /sicov-sync/diagnostico?placa=ABC123&nit=891100299

Retorna el histórico completo de alistamientos, preventivos y correctivos que SICOV tiene registrados para esa placa.

5.4 Migración de Fechas de Sincronización

Para registros creados antes de que se implementara el campo fechaSyncSicov, este endpoint rellena automáticamente ese campo usando la fecha de creación del registro:

POST /sicov-sync/migrate-sync-dates

Afecta todos los registros con sicov_sync_status: 'synced' que no tienen fechaSyncSicov. También está disponible desde el botón "Marcar fechas SICOV" en la vista de Alistamientos (cuando aplica).

Equivalente en MongoDB (ejecutar directamente en terminal):

db.enlistments.updateMany( { $or: [ { sicov_sync_status: { $exists: false } }, { sicov_sync_status: null }, { sicov_sync_status: 'synced', fechaSyncSicov: { $in: [null, undefined] } } ]}, [{ $set: { sicov_sync_status: 'synced', fechaSyncSicov: { $ifNull: ['$fechaSyncSicov', '$createdAt'] } }}] ) // Repetir para: preventive_details y corrective_details

5.5 Reparación de Transacciones Huérfanas

Ocasionalmente, SICOV puede haber recibido la primera fase del registro (apertura de mantenimiento) pero no la segunda (detalle). Esto genera una "transacción huérfana" que bloquea los reintentos.

El sistema detecta esto automáticamente durante los reintentos (respuesta 409 de SICOV) y lo corrige. Para una reparación masiva por placas específicas:

POST /sicov-sync/repair-orphans
{ "placas": ["ABC123", "XYZ789"], "tipoId": 3, "recordType": "enlistment", "vigiladoId": "891100299" }

Retorna el resultado por cada placa: repaired (huérfana recuperada), new (creada nueva) o error.

5.6 Estadísticas y Registros Pendientes

EndpointDescripción

6. API Keys para Integración Externa

6.1 ¿Qué son las API Keys?

Las API Keys permiten que sistemas externos (aplicaciones de terceros, ERPs, sistemas de flotas) envíen registros de mantenimiento a ProtegeMeSICOV de forma automática, sin necesidad de un usuario humano.

Se usan en el header HTTP X-Api-Key de las peticiones al endpoint de ingesta: POST /external-ingestion/maintenance.

6.2 Crear una API Key

POST /external-ingestion/api-keys
{ "name": "Sistema ERP Empresa XYZ", "enterprise_id": "id_de_la_empresa", "demoMode": false }

La respuesta incluye la API Key en texto plano. Guárdela inmediatamente, ya que no podrá recuperarla después (solo se almacena el hash SHA-256).

6.3 Modo Demo (demoMode)

Al crear una API Key con "demoMode": true, todos los registros enviados a través de esa clave se guardan localmente pero NO se envían a SICOV.

Los registros en modo demo quedan con:

  • sicov_sync_status: 'demo'
  • Sin entrada en la cola de reintentos.
  • Visible en la plataforma con el badge azul DEMO.
✅ Uso recomendado: Use demoMode para integraciones en fase de prueba, entornos de desarrollo o QA. Cuando la integración esté validada, cree una nueva API Key sin demoMode para producción.

6.4 Entorno Sandbox — Integración con Empresas Externas

Si una empresa externa (ERP, sistema de flotas, etc.) quiere integrarse con ProtegeMeSICOV para enviar alistamientos o mantenimientos en tiempo real o por lotes, puede habilitar un entorno de pruebas sin riesgo de enviar datos incorrectos a Supertransporte. Todo se guarda localmente con estado DEMO.

Paso 1 — Crear una empresa Sandbox (SuperAdmin)

En ADMINISTRACIÓN → Empresas → Nueva empresa cree una empresa de prueba:

CampoValor sugerido
NombreSandbox / Demo Integraciones
NITCualquiera (ej: 900000001)
VigiladoIdDejar vacío
Token VigiladoDejar vacío
EstadoActiva
ℹ️ Nota: No necesita credenciales SICOV reales. Con demoMode activado, ningún dato se envía a Supertransporte independientemente de las credenciales configuradas.

Paso 2 — Crear una API Key con demoMode

Con un usuario admin de esa empresa sandbox, realice la siguiente petición HTTP autenticada:

POST /external-api/keys Authorization: Bearer <jwt_token_del_admin_sandbox> Content-Type: application/json { "name": "Integración ERP Empresa XYZ", "demoMode": true }

La respuesta incluye la API Key en texto plano una sola vez:

{ "apiKey": "sk_live_abc123xyz...", "name": "Integración ERP Empresa XYZ", "demoMode": true }
⚠️ Guarde la API Key de inmediato. No se puede recuperar después; solo se almacena el hash en la base de datos.

Paso 3 — La empresa externa llama a su API

Comparta con la empresa integradora la URL base del sistema y la API Key. Deben incluir el header X-Api-Key en cada petición:

Registrar un alistamiento:

POST /external-api/enlistments X-Api-Key: sk_live_abc123xyz... Content-Type: application/json { "placa": "ABC123", "tipoIdentificacionConductor": 1, "numeroIdentificacionConductor": "12345678", "nombresConductor": "Juan López", "actividades": [1, 2, 3, 9, 10, 11] }

Registrar un preventivo:

POST /external-api/preventives X-Api-Key: sk_live_abc123xyz... Content-Type: application/json { ... misma estructura que el formulario web ... }

Registrar un correctivo:

POST /external-api/correctives X-Api-Key: sk_live_abc123xyz... Content-Type: application/json { ... misma estructura que el formulario web ... }

Paso 4 — Verificar los registros en la plataforma

Los registros enviados por la empresa externa aparecerán en la vista de Alistamientos / Preventivos / Correctivos de la empresa sandbox con el badge DEMO. Puede verificar que los datos llegan correctamente, el formato es válido y la integración funciona antes de pasar a producción.

Paso 5 — Pasar a producción

Cuando la integración esté validada y lista para producción:

1
Cree (o use) la empresa real con sus credenciales SICOV correctas (VigiladoId + Token Vigilado).
2
Cree una nueva API Key para esa empresa sin demoMode (o con "demoMode": false).
3
Entregue la nueva API Key a la empresa integradora para que actualice su configuración.
4
Revoque la API Key sandbox para evitar registros de prueba mezclados con los reales.

7.4 Cron Job

El cron de reintentos SICOV corre dentro del proceso NestJS de api-maintenance, sin necesidad de Redis, Bull ni workers externos. Se ejecuta automáticamente al iniciar el servicio. No requiere configuración adicional en producción.

⚠️ Importante: Si se reinicia el contenedor api-maintenance, el cron se reactiva automáticamente. No hay pérdida de reintentos ya que el estado persiste en MongoDB.

9. Resolución de Problemas Comunes

Una empresa no puede sincronizar con SICOV

Diagnóstico paso a paso:

1
Verifique en Auditoría SICOV el mensaje de error que devuelve Supertransporte en la columna "Mensaje Supertransporte".
2
Si el error menciona credenciales inválidas, revise el VigiladoId y el Token Vigilado en la configuración de la empresa (ADMINISTRACIÓN → Empresas → Editar).
3
Use GET /sicov-sync/diagnostico?placa=ABC123 para verificar si SICOV ya tiene registros para esa placa.
4
Si el problema persiste, verifique la conectividad del servidor con la URL de SICOV revisando los logs de Docker (docker logs api-maintenance).

Registros con estado "FALLIDO" que no se auto-recuperan

Si hay registros en estado failed que no se recuperan con el cron automático, puede deberse a transacciones huérfanas en SICOV. Use el endpoint:

POST /sicov-sync/repair-orphans

con las placas afectadas y el tipo de registro.

El filtro de fechas en Auditoría no devuelve resultados

Registros anteriores a la implementación del campo enterpriseId pueden no aparecer. Use el botón "Corregir registros históricos" en la vista de Auditoría SICOV. El sistema asignará la empresa a esos registros y aparecerán en los filtros.

El alistamiento aparece como "PENDIENTE" hace varios días

Posibles causas:

  • La API de SICOV estuvo caída o con problemas durante ese período.
  • Las credenciales de la empresa (VigiladoId/Token) estaban incorrectas.
  • La placa del vehículo no está registrada en SICOV maestras.

Fuerce el reintento inmediato con POST /sicov-sync/trigger-batch y revise la Auditoría para ver el error específico.

¿Cómo verificar que los datos llegaron a Supertransporte?

Use el endpoint de diagnóstico para consultar directamente en SICOV maestras:

GET /sicov-sync/diagnostico?placa=ABC123&nit=891100299

Si el registro aparece en el resultado, SICOV lo recibió correctamente. Si no aparece y el estado local es synced, use el endpoint de reparación de huérfanas.

📞 Soporte técnico especializado:
Para situaciones no cubiertas en este manual, el equipo técnico de ProtegeMeSICOV puede revisar directamente los logs del sistema, la base de datos y las comunicaciones con SICOV para un diagnóstico completo.
